# 前后端对接缺失汇总（2026-04-08）

## 2. 完成了什么（已对齐）

### 2.1 后端文档承诺接口：已落地
- 用户认证接口已存在：`/api/user-auth/register|login|refresh-token|me|logout`
- 用户资料接口已存在：`/api/users/me/profile`（GET/PUT）、`/api/users/me/avatar`（POST）
- 我的模块接口已存在：`/api/planes/mine/thrown|fueled|picked`
- 安全改造已存在：
  - `POST /api/planes/{id}/like` 需要登录，重复续航返回 409
  - `POST /api/planes/{id}/recall|destroy` 需要登录且作者本人
  - `POST /api/planes` 会写 `CreatorUserId`（有 app_user token 时）
  - `GET /api/planes/{id}`、`GET /api/planes/random` 会记录拾取（有 app_user token 时）

### 2.2 服务端可编译
- 已执行：`dotnet build -c Release`
- 结果：成功，0 error，0 warning

## 3. 未完成什么（当前缺失在用户端）

| 模块 | 当前状态 | 影响 | 原因 |
|---|---|---|---|
| 全局鉴权头注入 | `common/api.js` 的 `uni.request` / `uni.uploadFile` 未注入 `Authorization` | 新增鉴权接口会 401；“我的拾取/投递记录写入”失效 | 现有请求层是匿名请求 + mock 回退 |
| token 续期 | 无 `refresh-token` 自动续期与重放机制 | token 过期后业务中断，需要手动重登 | 目前没有会话层与拦截层 |
| 登录/注册 | `pages/login` / `pages/register` 仍调用 `common/auth.js` 本地账号逻辑 | 无法使用后端真实用户体系 | 历史实现为本地存储账号 |
| 退出与 me 校验 | 未调用 `/api/user-auth/me`、`/api/user-auth/logout` | 启动不校验远端会话；退出不撤销 refresh token | 仍沿用本地 session |
| 个人资料 | `pages/profile-edit` 明确“仅保存在本机”，调用 `setProfile*` 本地写入 | 资料不落后端，跨设备不同步 | 暂未接入 profile 接口 |
| 我的页数据源 | `pages/mine` 仍依赖 `myPlaneIds/fueledPlaneIds/recalledPlaneIds` + `POST /api/planes/mine` | “我的拾取”未接入，和服务端真实历史不一致 | 仍沿用本地 ID 列表方案 |
| 续航/回收/销毁前置登录 | 详情页可直接点续航/回收/销毁，未确保携带 token | 触发 401/403，用户感知为“接口异常” | 页面层未做鉴权门控 |

## 4. 为什么未完成（根因）
- 用户端当前架构是“本地账号 + 本地资料 + 本地我的记录 + 可回退 mock”，不是“后端会话驱动”。
- 新后端能力已经齐全，但用户端尚未完成会话层切换（token 存储、注入、续期、鉴权门控）。
- 页面业务状态仍以本地键为主数据源，导致与后端真实历史表（thrown/fueled/picked）未打通。

## 5. 需要怎么做（建议落地顺序）

### P0：先打通会话与鉴权
1. 新增统一会话存储（`accessToken`、`refreshToken`、`expiresIn`、`user`）。
2. 改造 `common/api.js`：
   - `uni.request`、`uni.uploadFile` 注入 `Authorization: Bearer {accessToken}`
   - 401 时尝试 `POST /api/user-auth/refresh-token`，成功后重放原请求
   - refresh 失败则清空会话并跳登录
3. 新增用户认证 API 封装：`register/login/me/logout/refreshToken`。

### P0：替换本地认证与资料逻辑
4. `pages/login` / `pages/register` 改为调用后端认证接口。
5. 应用启动或页面入口调用 `/api/user-auth/me` 校验登录态。
6. `pages/profile-edit` 改为：
   - 读：`GET /api/users/me/profile`
   - 写：`PUT /api/users/me/profile`
   - 头像：`POST /api/users/me/avatar`

### P1：替换“我的”页数据源
7. `pages/mine` 改为服务端列表：
   - 我的投递：`GET /api/planes/mine/thrown`
   - 我的续航：`GET /api/planes/mine/fueled`
   - 我的拾取：`GET /api/planes/mine/picked`
8. 本地 `myPlaneIds/fueledPlaneIds/recalledPlaneIds` 降级为短期缓存，不再作为主数据源。

### P1：页面行为与鉴权一致化
9. 在详情页/我的页中，对需要登录的操作（like/recall/destroy）先做登录态检查，未登录先跳登录页。
10. 投递、详情拾取、随机拾取尽量带 token 请求，以便后端写入 creator/picked 历史。

## 6. 快速验收清单（联调后）
- 注册 -> 登录 -> me -> 退出 全链路通过
- 资料修改后重新登录仍能读取到最新 profile/avatar
- 续航首次成功、二次返回 409
- 非作者回收/销毁返回 403，作者操作成功
- “我的投递/续航/拾取”三个列表与实际行为一致

---

结论：后端已基本按文档完成；当前主要缺口在用户端未切换到新接口体系。建议优先完成 P0（会话与认证），再推进“我的页”数据源替换。
